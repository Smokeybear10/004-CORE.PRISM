# Design: Attribution Evaluation Harness

Generated by /office-hours on 2026-04-24
Branch: scaffold-mentor-feedback
Mode: Builder (hackathon)
Status: DRAFT

## Problem Statement

We need a way to answer: **"did my prompt change make the attribution model
better, worse, or the same?"** Without a number after each change, prompt
iteration is vibes-based and the ablation chart in the demo has nothing real
behind it.

The harness must:
1. Run the attribution model on a fixed set of (ticker, date) test cases.
2. Compare each output to a hand-written expected answer.
3. Repeat across ablation configs so we see which data sources help.
4. Emit one JSON report that `demo/` reads to build the "accuracy vs sources"
   bar chart (the demo's money shot).

## Constraints

- **Hackathon scope.** MVP first, polish later. No CI, no fancy dashboards.
- **API cost.** Every `model.attribute()` call hits Claude. Cache aggressively.
  Cache key MUST include a prompt-version hash or prompt iteration is silently
  broken.
- **Schema in flux.** CLAUDE.md lists 5 dimensions (demand, pricing, competitive,
  management_credibility, macro) as fixed, but the team is not yet sure. Harness
  must survive dimension additions, removals, renames without rewrites.
- **Team can't realistically hand-author more than ~5 test cases.** Mentor rule:
  you must *know* the right answer. 5 cases > 10 noisy ones.

## Premises (all agreed)

1. One number per run. `composite` score = weighted sum of per-assertion matches.
2. Expected outputs are structured (dominant dim, direction, move_character) —
   never free-text prose. No LLM-grading-LLM.
3. Harness is a CLI runner + JSON report. One smoke pytest over cached outputs.
   No API calls in CI.
4. Cache key includes prompt-version hash. Tweak prompt → cache busts → re-scores.
5. 5-ish test cases, chosen because the team knows the cause.
6. **Flexibility over completeness.** Dimensions, weights, and which assertions
   are checked must be easy to change without touching scorer code.

## Approaches Considered

### Approach A: Pytest-only
Each fixture → one test function. Rejected: 5 cases × 6 ablations = 30 test
functions, no natural home for the demo JSON, CI would burn API credits.

### Approach B: CLI runner + JSON report (chosen)
Standalone `python -m eval.run` iterates the (case × ablation) matrix, scores
each output, writes `demo/eval_report.json`. Smoke pytest verifies scorer on
cached output. Demo chart reads JSON.

### Approach C: Notebook-driven
Jupyter interactive. Rejected: state drift, not reproducible, hard to diff
prompt-change effects, notebooks age badly during a 24h hackathon.

## Recommended Approach: B

Two nested loops, one JSON file out. Directly feeds the demo chart.

## Architecture

```
eval/
  __init__.py
  cases.py          # load_cases() -> list[ExpectedCase]
  scorer.py         # score(attribution, expected, config) -> ScoreResult
  runner.py         # run_matrix(cases, ablations) -> EvalReport
  cache.py          # cache_key(case, ablation, prompt_hash) + read/write
  config.py         # ScorerConfig: weights + which assertions to check
  cli.py            # python -m eval.run --cases all --ablations all

tests/
  fixtures/
    AAPL_2020-03-16_expected.json   # already exists — the frozen case
    <ticker>_<event>_expected.json  # 3-4 more to author
  test_scorer.py                    # unit tests, no API
  test_eval_smoke.py                # runs CLI over cached outputs

demo/
  eval_report.json                  # output — cells = (ablation, case) scores
  chart.py                          # plot bars per ablation
```

## Data Contracts

### Fixture format (dimension-agnostic)

Every assertion is **optional**. Scorer only scores what's present.

```json
{
  "ticker": "AAPL",
  "move_date": "2020-03-16",
  "known_cause": "COVID crash — market-wide panic, not AAPL fundamentals.",
  "expected": {
    "dominant_dimension": ["macro"],
    "direction": "negative",
    "move_character": "transient",
    "fade_or_lean": "fade",
    "must_cite_source_type": ["news", "macro"],
    "must_not_be_dominant": ["management_credibility"]
  }
}
```

Rules:
- `dominant_dimension` is a **list** so a fixture can accept either of two
  plausible top dimensions. Exact-match fixtures just use a one-element list.
- Any `expected.*` key missing = that assertion is skipped, not failed.
- `must_cite_source_type` supports the ablation logic: with `base_news` only,
  the model cannot cite `macro`, so it should either drop the assertion or
  score lower — the scorer handles this cleanly.
- String values (e.g. `"macro"`) are just strings compared to
  `Attribution` field names. No enum coupling.

### ScorerConfig

```python
@dataclass
class ScorerConfig:
    dim_weight: float = 0.5
    dir_weight: float = 0.25
    char_weight: float = 0.15
    fade_lean_weight: float = 0.10
    # assertions with weight 0 are computed-and-logged but excluded from composite
    # so you can A/B whether a metric matters without deleting data
```

### ScoreResult

```python
class ScoreResult(BaseModel):
    case_id: str                # f"{ticker}_{move_date}"
    ablation_name: str
    composite: float            # weighted sum, 0.0-1.0
    dim_match: Optional[bool]   # None if fixture didn't assert
    dir_match: Optional[bool]
    char_match: Optional[bool]
    fade_lean_match: Optional[bool]
    cited_source_types: list[str]
    notes: list[str]            # human-readable failures
```

### EvalReport

```python
class EvalReport(BaseModel):
    prompt_hash: str
    timestamp: datetime
    scores: list[ScoreResult]
    # aggregates computed at write time for cheap chart rendering
    composite_by_ablation: dict[str, float]       # ablation_name -> mean composite
    dim_accuracy_by_ablation: dict[str, float]    # ablation_name -> % cases with dim_match
```

## The Flex: How Dimension Changes Don't Break the Scorer

**Scorer finds dimensions via schema introspection, not a constant.**

```python
# scorer.py
def _dimension_fields(attribution: Attribution) -> dict[str, DimensionScore]:
    """Return every DimensionScore-typed field, whatever it's called."""
    result = {}
    for name, field_info in Attribution.model_fields.items():
        value = getattr(attribution, name)
        if isinstance(value, DimensionScore):
            result[name] = value
    return result

def _dominant_dimension(attribution: Attribution) -> str:
    dims = _dimension_fields(attribution)
    return max(dims, key=lambda name: dims[name].weight)
```

Consequences:
- Drop `management_credibility` from `Attribution` → scorer still works, any
  fixture referencing it will just fail that assertion (clear error message).
- Rename `pricing` → `pricing_power` → update fixtures that use the old name,
  scorer code untouched.
- Add `regulatory` as a 6th dimension → scorer picks it up automatically.
- Cut to 3 dimensions → scorer doesn't care how many there are.

Schema changes still require team chat per CLAUDE.md. But the *harness* doesn't
care, and fixture edits are localized.

## Caching

Cache key:
```python
key = sha256(f"{ticker}|{move_date}|{ablation_name}|{prompt_version_hash}")
```

`prompt_version_hash` = sha256 of the prompt template string in
`model/prompts.py` (or wherever it lives). Any whitespace change busts the
cache, which is what you want — you must not trust old scores after a prompt
edit.

Cache dir: `.cache/eval/{key}.json`, containing the raw `Attribution` JSON.

CLI flag `--no-cache` for debugging.

## CLI Interface

```
python -m eval.run                      # all cases × all ablations
python -m eval.run --cases AAPL_2020-03-16
python -m eval.run --ablations base_news,+sec
python -m eval.run --no-cache
python -m eval.run --report demo/eval_report.json    # default path
```

Output:
- Writes `demo/eval_report.json`.
- Prints per-ablation composite to stdout so you see the number without
  opening the file.
- Exit code: 0 always (it's a measurement, not a gate). Upgrade to non-zero
  exits via a `--gate` flag later if we want CI-style regression blocking.

## Open Questions

1. **Do we care about the `move_character` assertion during MVP?** If the model
   is still flaky on it, including it in the composite tanks every score and
   makes bar-chart differences hard to read. Default: include with
   `char_weight=0.15`, but document that dropping it to 0 is a legitimate move.

2. **How do we handle fixtures where `must_cite_source_type` includes sources
   the current ablation excludes?** Options: (a) auto-skip that assertion per
   ablation, (b) score as fail (penalizes low-source ablations fairly), (c)
   score as NA and log. Recommend (a) for MVP — otherwise `base_news` always
   looks artificially bad on macro-driven cases.

3. **Do we want a "null ablation" (empty sources)?** Useful sanity check:
   model should output low confidence when it has nothing. Low priority.

4. **Fixture authoring workflow.** Mentor's rule: write expected output BEFORE
   running the model. In practice, people will peek at model output and call
   it the expected. Mitigation: fixture files go through PR review with a
   teammate who hasn't seen the model run yet.

## Success Criteria

- `python -m eval.run` produces `demo/eval_report.json` without errors on the
  current AAPL frozen case.
- Changing a word in the attribution prompt produces a different composite
  score (cache busts correctly).
- Adding a dimension to `schema.Attribution` requires zero edits to `scorer.py`.
- The demo chart can be built from `eval_report.json` alone — no re-running
  the model at chart time.

## Next Steps (concrete, in order)

1. **Author 3-4 more fixture files.** Pick (ticker, date) pairs where a
   teammate knows the cause. Format per `tests/fixtures/AAPL_2020-03-16_expected.json`.
2. **`eval/config.py` + `eval/scorer.py`.** Start with dimension introspection
   helper + dominant-dim scoring. Unit test against fake `Attribution`
   instances (no API).
3. **`eval/cache.py`.** Prompt-hash-keyed cache. Write tests before runner.
4. **`eval/runner.py` + `eval/cli.py`.** Glue: load cases, loop ablations, call
   `backtest.run_ablation` (already stubbed), score, write report.
5. **`demo/chart.py`.** Read `eval_report.json`, plot per-ablation composite
   bars and per-dimension-accuracy secondary bars.
6. **Smoke pytest.** Runs the CLI over cached outputs, asserts report parses
   and contains expected ablation names.

## Non-Goals (explicit)

- Grading free-text rationale.
- CI integration.
- Multi-ticker universe beyond the frozen test cases.
- Trading performance (that's `backtest.evaluate()`, separate concern).
- Automated prompt optimization.
